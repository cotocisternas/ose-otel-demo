## Step 16: Head Sampling + W3C Baggage

Two sub-lessons under one git tag. Sub-lesson 16a shows the SDK dropping traces *before*
export. Sub-lesson 16b shows a business property (`customer-tier`) flowing from an HTTP
header through AMQP into both producer and consumer spans automatically.

### Step 16a: Head Sampling

> **Warning — Double-filter trap (F2-3):** Phase 11 tail sampling (20% probabilistic
> fallback at the Collector) is still active if you left it on. Activating Phase 16 head
> sampling (50% at the SDK) **compounds** those two rates: **50% × 20% = ~10% effective
> trace rate**. Send 100 requests — expect ~10 traces in Tempo, not ~50. To isolate the
> pure head-sampling effect, temporarily disable Phase 11's `probabilistic-fallback` policy
> while running this lesson. To disable: set `OTEL_TRACES_SAMPLER` has **no effect** here
> (the demo uses manual SDK construction without autoconfigure — see Pitfall F7-1). Instead,
> comment out the `probabilistic-fallback` policy block in `otelcol-config.yaml` and restart
> the Collector.

#### What you'll learn

- **Where the sampling decision happens.** The SDK decides at span *start* — before the span
  is ever exported to the Collector. The Collector never receives the dropped spans. Compare
  this to Phase 11 tail sampling: the Collector received every span and made its decision
  *after* the trace was fully assembled.

- **Head vs tail contrast.** The table below captures the five decisive dimensions:

  | Dimension | Head sampling (SDK) | Tail sampling (Collector) |
  |-----------|--------------------|-----------------------------|
  | **Where** | Inside `SdkTracerProvider` at span start | Inside Collector `tail_sampling` processor |
  | **Sees** | Only the root span's trace ID — no downstream spans exist yet | The entire assembled trace (error status, total latency, all spans) |
  | **Bandwidth** | Drops spans before OTLP export — saves network + storage | All spans exported to Collector; sampling happens at storage time |
  | **Decides on** | A hash of the trace ID (probabilistic, no trace context) | Full trace attributes (error codes, p99 latency, service names) |
  | **Trade-off** | Cheap; cannot preserve 100% of errors unless you use `parentBased` | Expensive (all spans traverse network); can keep 100% of errors |

- **The `parentBased` wrapper is load-bearing.** `Sampler.traceIdRatioBased(0.5)` alone would
  re-roll a new sampling decision per span — you would see trace *fragments* where the consumer
  span is sampled but the producer parent is not. `Sampler.parentBased(...)` wraps it: if the
  incoming `traceparent` flag byte says "sampled", all downstream spans are sampled too. All spans
  within a trace are sampled or dropped together.

- **What changed in the code.** One line per service in `OtelSdkConfiguration.buildTracerProvider()`:
  ```java
  // Phase 16a — HSAMP-01: swapped from alwaysOn() to 50% ratio sampling
  Sampler sampler = Sampler.parentBased(Sampler.traceIdRatioBased(0.5));
  ```
  No Collector config change. No `OTEL_TRACES_SAMPLER` env var. The SDK is wired
  programmatically; env vars are silently ignored (no autoconfigure on classpath).

### Step 16b: W3C Baggage

#### What you'll learn

- **Baggage is W3C-standardised cross-cutting context.** The W3C Baggage specification defines a
  `baggage` header (`baggage: key=value,key2=value2`) that propagators inject and extract alongside
  `traceparent`. Because `W3CBaggagePropagator` is already in the composite propagator chain (wired
  in Phase 2), the `baggage:` header flows through AMQP and HTTP *automatically* — no changes to
  `TracingMessagePostProcessor` or `TracingClientHttpRequestInterceptor` are needed.

- **Three coordinated changes make baggage visible in Tempo.** Baggage is set (controller → `Baggage.builder().put()`), stamped (processor → `BaggageSpanAttributeProcessor.onStart()`), and made available to the consumer body (listener advice → `extracted.makeCurrent()`). All three must be active for `baggage.customer-tier` to appear on both sides of the AMQP boundary.

- **The `BaggageSpanAttributeProcessor` pattern.** A `SpanProcessor` that reads from `parentContext`
  (not from `Baggage.current()`) and stamps only an explicit allowlist of keys as `baggage.<key>`
  span attributes. The allowlist (`Set.of("customer-tier")`) is the cardinality guard — arbitrary
  upstream baggage never becomes unbounded high-cardinality Tempo attributes.

- **The SERVER span does not carry `baggage.customer-tier` — and that is intentional (D-B5).**
  `HttpServerSpanFilter` creates the SERVER span *before* `OrderController.create()` runs, before
  baggage is set. `BaggageSpanAttributeProcessor.onStart()` fires with empty baggage for the SERVER
  span → no attribute. The INTERNAL, PRODUCER, CLIENT, and CONSUMER spans (all created *after*
  `baggage.makeCurrent()`) carry `baggage.customer-tier`. The trace waterfall in Tempo visualises
  baggage's lifecycle: it is not retroactive — it stamps only spans started within its active scope.

- **What changed in the code (three files):**
  1. `otel-bootstrap/context/BaggageSpanAttributeProcessor.java` — new `SpanProcessor` (BAG-02)
  2. `OrderController.create()` — reads `X-Customer-Tier` header, wraps `orderService.place()` in
     `try (Scope baggageScope = baggage.makeCurrent())` (BAG-01)
  3. `TracingMessageListenerAdvice.invoke()` — outer `try (Scope ctxScope = extracted.makeCurrent())`
     wraps the entire listener body so `Baggage.current()` is populated throughout (BAG-03)

**Both the PRODUCER-side and CONSUMER spans carry `baggage.customer-tier=gold`.** Open the trace in
Tempo. Expand the span details:

```

        ← baggage.customer-tier=gold  ✓

        │     ← baggage.customer-tier=gold  ✓ (on the same producer trace)
        └── CLIENT span (TracingClientHttpRequestInterceptor)
              ← baggage.customer-tier=gold  ✓

[separate consumer trace — linked by traceparent in AMQP baggage header]
  └── CONSUMER span (TracingMessageListenerAdvice — orders.direct process)
        ← baggage.customer-tier=gold  ✓ (survived the AMQP boundary)
```

Notice: the SERVER span has no `baggage.customer-tier` — it was started by `HttpServerSpanFilter`
before your controller ran. This is not a bug; it is the baggage lifecycle made visible.
