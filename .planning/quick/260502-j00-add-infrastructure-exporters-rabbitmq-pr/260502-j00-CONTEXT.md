# Quick Task 260502-j00 — Step 9: Dashboards & Infrastructure Telemetry — Context

## Task Boundary

<domain>

Workshop pedagogy. Phase 8 added Valkey + PostgreSQL, with manual OTel SDK instrumentation in the apps (Jedis SET spans, JDBC INSERT spans, HikariCP gauge). The current Grafana dashboards (`ose-otel-demo.json`, `ose-otel-noc.json`) were authored during Phase 7 and do **not** render any of that new telemetry. Workshop attendees who run the Phase 8 checkpoint cannot *see* what they just instrumented unless they hand-author Tempo/Mimir/Loki queries — defeating the demo's "zero clicks, see your telemetry" promise.

Beyond catching dashboards up to Phase 8, this task adds a **second telemetry source**: dedicated infrastructure exporters (RabbitMQ Prometheus plugin, `oliver006/redis_exporter`, `prometheuscommunity/postgres-exporter`). Together with the existing app-emitted OTel data, this teaches attendees the **two-lens model** that real production observability requires:

- **App-emitted telemetry** answers: "What did *my code* do to this dependency?" (spans + business metrics)
- **Infra exporters** answer: "What is *the dependency itself* doing?" (queue depth, eviction rates, slow queries, replication lag, table bloat)

Neither lens alone is sufficient in production. The workshop's final chapter should make this lesson explicit.

</domain>

## Specific Ideas

<specifics>

- **Existing dashboards** (workshop + NOC) MUST remain functionally unchanged. The 3rd dashboard supplements; it does not replace.
- **Auto-provisioning** is already wired via `grafana/dashboards/dashboards.yaml` — drop the new JSON in the same dir and it's picked up. No new provisioning manifest needed unless researcher finds a reason.
- **RED Metrics on existing dashboards** are powered by Tempo's metrics-generator (Phase 7). The new exporters feed Mimir directly via Prometheus scrape — different data path. Dashboards must use the right datasource UID per panel.
- **HikariCP gauge** (`db.client.connection.count`) is already emitted by `consumer-service` via `HikariCpConnectionGauge`. Should appear in the Database row of the new dashboard alongside `postgres_exporter`'s connection-count metrics — teaches that the *same metric name* can come from two different sources, and how to disambiguate via the `service.name` resource attribute.
- **Idempotency hit/miss** is currently visible only via Valkey spans in Tempo (`db.system.name=redis`, `db.statement` showing `SET ... NX EX`). Could be turned into a Prometheus counter (TraceQL → metrics-generator?) OR derived from `redis_exporter`'s `redis_commands_processed_total{cmd="set"}`. Researcher to surface trade-offs.

</specifics>
